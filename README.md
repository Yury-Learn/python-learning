## Структура репозитория

Планируемая структура (после небольшого рефакторинга):

- `week1/`
  - `week1_day1.py` — первый скрипт и базовый синтаксис Python.
  - `week1_day2.ipynb` — первый Jupyter Notebook с экспериментами.
  - `week1_day3.py` — ввод данных (`input()`), простая логика.
  - `week1_day4.py` — списки, словари, цикл `for`, подготовка к «админским» задачам.
  - `day1_plan.md` — план на первый день (что делать, что спрашивать у ChatGPT, какой код писать).
  - `day2_plan.md` и т.д. (появятся позже).

- `week2/` (появится позже)  
  Работа с файлами (логи), `sys.argv`, простые консольные утилиты администратора.

- `.gitignore`  
  Файл, который говорит Git, какие файлы не надо коммитить (кэш интерпретатора Python, временные файлы, виртуальные окружения и т.д.).

## Рабочая среда

Я использую:

- **Python 3** — основной интерпретатор (желательно 3.10+).
- **VS Code** — IDE, в которой:
  - открываю этот репозиторий;
  - запускаю `.py`-файлы;
  - работаю с Jupyter Notebook.

- Расширения VS Code:
  - `Python` (Microsoft);
  - `Jupyter` (Microsoft);
  - Git-инструменты (по желанию).

- **Git + GitHub**:
  - локально работаю с кодом;
  - коммичу изменения;
  - пушу в репозиторий `Yury-Learn/python-learning`.

- При необходимости использую виртуальное окружение:

```bash
python3 -m venv .venv
source .venv/bin/activate      # Linux/WSL
# .venv\Scripts\activate.bat # Windows (cmd)
# .venv\Scripts\Activate.ps1 # Windows (PowerShell)
```

`.venv` добавляется в `.gitignore`, чтобы не засорять репозиторий.

## Как я запускаю скрипты

Пример:

```bash
python3 week1/week1_day1.py
```

Где:

- `python3` — интерпретатор Python.
- `week1/week1_day1.py` — мой учебный скрипт за первый день.

Jupyter-ноутбуки (`.ipynb`) я открываю прямо в VS Code:

1. Открываю файл, например `week1/week1_day2.ipynb`.
2. Выбираю интерпретатор Python (обычно виртуальное окружение `.venv`, если оно есть).
3. Запускаю ячейки по очереди и экспериментирую с кодом.

## Как я использую ChatGPT

Я не прошу «сделай за меня весь код».

Вместо этого я прошу ChatGPT:

- объяснить, что делает каждая строка моего кода;
- помочь понять ошибку и причину, а не просто дать готовый фикс;
- подсказать, как структурировать проект (имена файлов, функции, аргументы);
- предложить более аккуратный вариант решения, чтобы я понимал, что именно изменилось.

Цель — не только «код работает», а «я понимаю, что я делаю».

## Стиль кода в этом репозитории

Чтобы код выглядел привычно для других инженеров:

1. **Имена файлов**
   - Учебные дни: `week1_day1.py`, `week1_day2.ipynb` и т.д.
   - Внутри папок используются осмысленные названия: `day1_plan.md`, `day2_plan.md`.
   - Только латиница и нижний регистр, без пробелов.

2. **Имена переменных**
   - `snake_case`: `server_list`, `host_name`, `error_count`.
   - Название отражает смысл:
     - `servers` — список серверов;
     - `server` — один сервер;
     - `log_line` — строка лога.

3. **Имена функций**
   - `snake_case`, глагол + существительное:
     - `read_log_file()`;
     - `parse_log_line()`;
     - `print_inventory_table()`.
   - Функция отвечает за одну конкретную задачу.

4. **Комментарии и язык**
   - Код (имена переменных/функций) — по-английски.
   - Комментарии и Markdown-документация — по-русски.
   - Комментирую:
     - неочевидную логику;
     - привязку к админской реальности (формат логов, особенности хостов и т.п.).

5. **Форматирование**
   - Отступы — 4 пробела.
   - Стараюсь не делать строки слишком длинными (примерно до 79–100 символов).
   - Кавычки — единый стиль внутри файла (`"` или `'`).
